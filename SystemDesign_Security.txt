############################################################ 1.Authentication and Authorization (Alina Kovtun) ###############################################################

Authentication is the process of verifying the user's identity.
In simple terms Authentication answers the question "Who are you".

In most applications like Facebook or LinkedIn, every user has a unique ID. When you try to log in, the system must confirm that you are the actual owner of that ID.
To prove identity, users provide evidence called authentication factors.
Below are the types of Authentication.

1. Knowledge-Based Authentication :
-----------------------------------
This is the most common method.
Examples :
Password.
PIN.
Security questions.

✔ Advantage: Easy to use.
✘ Disadvantage: Not very secure if credentials are stolen.

2. Possession-Based Authentication :
------------------------------------
Here Authentication is based on something we have.
Examples :
OTP via SMS or phone call.
Key cards or badges.
Authenticator apps / token devices.

3. Multi-Factor Authentication (MFA) :
--------------------------------------
This combines multiple methods for better security.
Example :
Password (something we know) + OTP (something we have).
✔ More secure because even if one factor is compromised, the system is still protected.

Authorization happens after the identity is confirmed. Authorization answers the question "What are you allowed to do".
Authorization is like Yes or No.
Once the user is successfully Authenticated, What the user can do depends on who the user is called Authorization.
Authentication and Authorization are widely used in the Software Development.

Note :
------
Always remember "N before Z" i.e. we cannot authorize someone if we don’t know who they are.

Secure Login Flow :
-------------------
1.User sends the credentials over HTTPS.
2.Server finds the user in the database by their email/username.
3.Server takes the provided password, processes it through a hashing algorithm and compares it with the stored version.
4.If the credentials matches then the server generates a "proof of entry" i.e. usually a SessionID or JWT and sends it back to the client.

Never Encrypt Passwords :
-------------------------
Encrypt the passwords in the database are technically wrong and that’s dangerous.
Encryption is a two-way street and if we encrypt the password then we can decrypt it.
There will be a secret key somewhere on the server.
If someone finds the key then it is possible to reverse every single password in the database.

Hashing is one way. 
A cryptographic hash function that takes an input i.e. the password and turns it into a fixed-length string of characters i.e. hash.
Mathematically we cannot reverse the hash to find the original password.

Rainbow Table with Salts :
--------------------------
Consider the scenario where two users use the same password then their hashes will be identical.
Hackers are aware of this.
They use Rainbow Tables i.e. massive lists of pre computed hashes for millions of common passwords.
If they see the hash in the database that matches with their pre computed hash then the password is cracked instantly.
To resolve this we use Salt.

A salt is a random string of data added to the password before it is hashed.
This ensures that even if two users have the same password, their hashes will look completely different.
There is no need to generate our own salts or crypto functions.
Use industry-standard algorithms that handle salting for us.

Preferred Algorithms :
----------------------
Not all hashes are created equal.
Some are too fast.
If an algorithm is fast then the hacker can guess billions of passwords per second.
For example MD5 or SHA-256.
We require an algorithm that is computationally expensive i.e. intentionally slow to make brute-force attacks impossible.

	| Algorithm  | Speed               |Salt Handling | Recommended       |
	|------------|---------------------|--------------|-------------------|
	| MD5 / SHA-1| Extremely Fast      | Manual       | NO (Broken)       |
	| SHA-256    | Fast                | Manual       | NO (Too fast)     |
	| bcrypt     | Slow (Adjustable)   | Automatic    | YES               |
	| Argon2id   | Memory-Hard         | Automatic    | YES (Best)        |

############################################################ 2.Role-Based Access Control (RBAC) ###############################################################

Effectively managing user access and permissions is a fundamental requirement for any computing system.
Once users have logged in, the system must control what they are allowed to access based on who they are.
This is where the authorization comes into play.
Every user has a unique identity like user ID, role or attributes.
Based on this identity, the system assigns permissions.
These permissions define which resources the user can access and what actions they can perform (read, write, delete, etc.)

RBAC is a security model where users are given access to resources based on their roles i.e. job responsibilities instead of assigning permissions individually.
After a user logs in, System needs a structured way to decide what they can access.
RBAC solves this by grouping permissions into roles.
    
Instead of assigning permissions to each user separately
Permissions are grouped into roles.
Users are assigned to those roles.

Here access control becomes

	User → Role → Permissions → Resources.
	
1.Role :
--------
Role represents the specific job or responsibility within an organization.
Roles are like categories or labels that groups the individuals based on their tasks.

Examples :
Admin
Manager
Employee

Each role has a defined set of responsibilities.

2.Permissions :
---------------
Permissions are the set of rules for a role that defines what actions or operations are allowed to perform on certain resources or data.

Examples :
Read data
Write data
Delete records
Update settings

3.Users :
---------
Users are individuals or entities within the organization.
Users are assigned to one or more roles based on their job.

RBAC Advantages :
-----------------
No need to manage permissions for each user.
Changes are done at role level.
Users get only required access.
Avoids manual mistakes in assigning permissions.

Types of RBAC :
---------------
RBAC can be categorized into several types or models, each with its own approach to manage user access based on roles.
The three primary types of RBAC are
1.Flat RBAC
2.Hierarchical RBAC
3.Constrained RBAC

1.Flat RBAC :
=============
Flat RBAC is the simplest form of RBAC where users are assigned roles and roles have permissions but there is no hierarchy between roles.

Roles are created independently.
Each role has its own permissions.
Users are assigned one or more roles.
Roles do not inherit permissions from other roles.
Every role is completely separate.

Characteristics :
-----------------
No parent-child relationship between roles.
No inheritance of permissions.
Simple mapping of users to roles.
Easy to implement and understand.
Suitable for small systems.

Consider a small company system.
Roles are Admin, Editor and Viewer.

Assign Permissions :
Admin can Read data, Write data and Delete data.
Editor can Read data and Write data.
Viewer can Read data.

Assign Users :
John → Admin
Priya → Editor
Ravi → Viewer

Here Editor role does not inherit anything from Admin.
Viewer role does not inherit from Editor.
Even if permissions look similar they must be defined separately for each role.

Practical Problem in Flat RBAC :
--------------------------------
Suppose if we want to add new permission to Download Report.
Now we must manually update Admin,Editor and Viewer role.
This becomes repetitive and error-prone.

Limitations :
-------------
No reuse of permissions.
Hard to manage as system grows.
No hierarchy leads to duplication.

2.Hierarchical RBAC :
=====================
Hierarchical RBAC is an advanced RBAC model where roles are arranged in a hierarchy (tree structure) and higher-level roles automatically inherit permissions from lower-level roles.
A higher role gets all the permissions of the lower role plus its own additional permissions.
Here Roles are connected like a tree.

	Admin
	  ↑
	Manager
	  ↑
	Employee
	
Employee → Basic permissions.
Manager → Employee permissions + extra.
Admin → Manager + Employee permissions + full control.

Consider the scenario of company system.

Roles :
Employee (lowest level)
Manager (middle level)
Admin (highest level)

Permissions :
-------------
Employee :
View profile
Submit requests

Manager :
Approve requests
View team data

Admin :
Delete users
Manage system

After Inheritance :
-------------------
Employee :
View profile
Submit requests

Manager :
View profile (inherited)
Submit requests (inherited)
Approve requests
View team data

Admin :
All Employee permissions
All Manager permissions
Delete users
Manage system

Here no need to redefine permissions again and again. If we update Employee permissions then Manager and Admin are automatically get updated.

Advantages :
------------
1.Reduces Duplication.
2.Easier Maintenance.
3.Matches Real Organizations.

3.Constrained RBAC :
====================
Constrained RBAC is an extension of RBAC where extra rules are applied to control how roles and permissions are assigned and used.
In basic RBAC, users get access based only on roles. But in real-world systems, this is often not enough.
Sometimes access must depend on conditions or rules not just roles. Access is allowed only if certain conditions are satisfied.

	User → Role → Permissions + Constraints
	
Even If a user has a role, they may not be allowed to use it unless conditions are met.

Types of Constraints :
1.Separation of Duties (SoD)
2.Cardinality Constraints
3.Prerequisite Constraints
4.Temporal Constraints

1.Separation of Duties (SoD) :
------------------------------
Prevents a single user from having conflicting roles.
A user cannot be both Payment Creator and Payment Approver.
This prevents the fraud.

2.Cardinality Constraints :
---------------------------
Limits number of users in a role.
Only 1 Admin allowed.
Only 2 Finance Heads allowed.

3.Prerequisite Constraints :
----------------------------
User must satisfy a condition before getting a role.
To become Manager User must first be Employee.
To access system User must complete training.

4.Temporal Constraints :
------------------------
Access allowed only during certain time.
Office system accessible only from 9 AM – 6 PM.
Admin access allowed only during working hours.

############################################################ 4.OAuth 2.0 (Vinotech) ################################################################################

OAuth 2.0 i.e. Open Authorization 2.0 is an open standard protocol for Authorization.
OAuth 2.0 allows the applications (clients) to access resources (like user data) from a server (resource server) on behalf of a user (resource owner).
OAuth 2.0 provides a way for users to grant limited access to their resources without sharing their credentials with third-party applications.

OAuth 2.0 is widely used to enable secured delegated access which allows different applications to interact with user data across different services.
Here the access is Limited,Scoped,Temporary and Revocable.
It is particularly common in scenarios involving social media logins, API access and mobile applications.

Components of OAuth 2.0 :
-------------------------
1.Resource Owner
2.Authorization Server
3.Client
4.Resource Server

1.Resource Owner :
------------------
The user or entity that owns the data and can grant access to it.

2.Authorization Server :
------------------------
The server that authenticates the resource owner and issues access tokens to the client after obtaining the consent.

3.Client :
----------
The application which is requesting an access to the resource owner’s data.

4.Resource Server :
-------------------
The Server that hosts the resource owner data and accepts the access tokens for granting the access to the data.

OAuth2 Shopping Cart Example Using Amazon :
===========================================
Consider the scenario of third-party application called QuickCart App.
QuickCart App helps to compare products, create smart shopping carts, and perform one-click checkout using Amazon account data.
QuickCart needs the permission to access Amazon instead of asking the user for username/password directly.

	Amazon profile
	Saved delivery addresses
	Saved payment methods
	Amazon cart/wishlist data
	
Here QuickCart uses OAuth2.

	| OAuth2 Component     | Amazon Example                |
	| ---------------------| ------------------------------|
	| Resource Owner       | Amazon customer               |
	| Client               | QuickCart App                 |
	| Authorization Server | Amazon Authorization Server   |
	| Resource Server      | Amazon APIs                   |
	| Access Token         | Permission token              |
	| Refresh Token        | Used to get new access tokens |
	
High level objective :
----------------------
User wants Quick checkout without entering address/payment repeatedly.
OAuth2 allows QuickCart to securely access Amazon user data without sharing Amazon password.

Step 1 : User Opens QuickCart App :
-----------------------------------
User browses the products in QuickCart.
At checkout QuickCart says Connect to your Amazon account for faster checkout.
User clicks Login to Amazon.

Step 2: QuickCart Redirects the User to Amazon Authorization Server :
---------------------------------------------------------------------
QuickCart redirects the browser to

	GET https://auth.amazon.com/oauth2/authorize
	
Complete CURL request.

	GET https://auth.amazon.com/oauth2/authorize?
	response_type=code
	&client_id=quickcart-client-101
	&redirect_uri=https://quickcart.com/oauth/callback
	&scope=profile.read address.read payment.read
	&state=abcxyz123
	
	| Parameter          | Meaning                 |
	| ------------------ | ------------------------|
	| response_type=code | Authorization Code Flow |
	| client_id          | Identifies QuickCart    |
	| redirect_uri       | Callback URL            |
	| scope              | Requested permissions   |
	| state              | CSRF protection         |

Step 3: Amazon Shows Login Page :
---------------------------------
Amazon asks user to login with Email and Password.
Here QuickCart doesn't have an access to Amazon credentials and is directed to Amazon.

Step 4: Amazon Shows Consent Screen :
-------------------------------------
Amazon displays QuickCart wants permission to Allow / Deny.

	✔ Read profile
	✔ Read saved addresses
	✔ Read payment methods

User clicks Allow.

Step 5: Amazon Generates Authorization Code :
---------------------------------------------
Amazon generates the Authorization Code and redirects the browser back to QuickCart.

	GET https://quickcart.com/oauth/callback?code=AUTH_CODE_12345&state=abcxyz123
	
Here Authorization code is Short-lived,One-time use and Temporary.

Step 6: QuickCart Exchanges Authorization Code for Access Token :
-----------------------------------------------------------------
Now QuickCart backend contacts Amazon backend directly.

	POST https://auth.amazon.com/oauth2/token
    Content-Type: application/x-www-form-urlencoded
	
	grant_type=authorization_code
	code=AUTH_CODE_12345
	client_id=quickcart-client-101
	client_secret=quickcart-secret
	redirect_uri=https://quickcart.com/oauth/callback

Step 7: Amazon Returns Access Token :
-------------------------------------

	{
	  "access_token": "ACCESS_TOKEN_ABC123",
	  "refresh_token": "REFRESH_TOKEN_XYZ789",
	  "token_type": "Bearer",
	  "expires_in": 3600,
	  "scope": "profile.read address.read payment.read"
	}
	
	| Token         | Purpose                   |
	| ------------- | ------------------------- |
	| Access Token  | Access Amazon APIs        |
	| Refresh Token | Generate new access token |
	| Expiry        | Security purpose          |
	
Step 8: QuickCart Calls Amazon APIs :
-------------------------------------
QuickCart can access Amazon APIs using the token.

Fetch User Profile.
Fetch Saved Addresses.
Fetch Saved Payment Methods.

	GET https://api.amazon.com/user/profile
	Authorization: Bearer ACCESS_TOKEN_ABC123
	
	GET https://api.amazon.com/user/addresses
	Authorization: Bearer ACCESS_TOKEN_ABC123
	
	GET https://api.amazon.com/user/payment-methods
	Authorization: Bearer ACCESS_TOKEN_ABC123
	
Step 9: Amazon Resource Server Validates Token :
------------------------------------------------
Before returning the data Amazon checks

	Was token issued by Amazon?
	Is token expired?
	Does token contain required scope?
	
For profile API, Required scope = profile.read
For payment API, Required scope = payment.read

Step 10: QuickCart Shows Personalized Checkout :
------------------------------------------------
Now QuickCart can show the below details without asking the user to re-enter details.

	Welcome Srinivas
	Select delivery address
	Select saved payment method
	Place Order

############################################################## 5.OpenID Connect(James Collerton) ###########################################################################

OAuth2.0 is about Authorization and never guarantees who the user is.
OAuth2.0 is about getting an access token with the right scopes that can be presented to get protected resources.
OAuth2.0 refers to

	Can this app access your data (or) This client is allowed to access this resource.

OpenId Connect is about Authentication and Authorization and refers to
	
	Who are you

OpenId Connect is an identity layer built on top of OAuth 2.0 that allows applications to verify who the user is (authentication) and get basic user profile information. 
OpenId Connect is about getting an Id token that we can present to the sites to prove the identity.

Why OIDC was introduced :
-------------------------
OAuth 2.0 was designed for authorization only but developers started misusing it for login.
OAuth2 does not standardize user APIs.

Problem:
OAuth gives the Access Token but
1.No standard way to identify the user.
2.No guarantee who the user is.
3.Different providers return different data formats.
Google uses /userinfo.
GitHub /user.
Facebook /me.
OIDC standardizes /userinfo.
OIDC defines standard claims i.e. sub,email,name,picture etc.

Solution:
OIDC adds a standard identity layer.

Usecase :
---------
Consider the scenario where we loggedIn to Medium using Google login.
Here we used OIDC.
Google acted as an Identity provider (IDP), issuing an ID token, which Medium used to check you are who you say you are.

	User
	  ↓
	Client App
	  ↓
	Identity Provider
	  ↓
	Authorization Code
	  ↓
	Access Token + ID Token
	  ↓
	Validate ID Token
	  ↓
	User Authenticated

OIDC standardizes the result of an Authentication but not the authentication method itself.
Whether they use a username and password, an authentication code or any other.

ID token :
----------
ID token is a JWT Token containing a set of specific claims that can be used to tell the people about the user Identity.
These include things like name, email address and birthday.

	{
	  "sub": "12345",
	  "name": "Srinivas",
	  "email": "user@gmail.com",
	  "iss": "https://accounts.google.com",
	  "aud": "client-id"
	}

We send a request to our IDP asking for an ID token.
This takes the form of an authorisation request which is exactly as OAuth2.0 but with the openId scope.
The IDP does the work to perform Authentication and Authorisation from the end user, redirecting back to the party requesting an ID token.
This response will provide both Access token, ID token and optionally refresh tokens.
This is opposed to OAuth2.0 which only issues access and refresh tokens.

Finally UserInfo endpoint is a way of getting a user’s information from their Access token.
We send a request with the token and the endpoint sends back a JSON list of user details.

The more complex part of the endpoint is how the IDP Authorisation Server knows what information we are allowed to see.
When we ask an end user for Authorisation, there are a number of custom OIDC scopes we need to request.
These scopes map to the standard claims for the ID token JWT.
For example if the user grants us the email scope, we will at least see their email and email_verified information in the response from the UserInfo endpoint.

	| Feature        | OAuth 2.0       | OpenID Connect                 |
	| -------------- | --------------- | ------------------------------ |
	| Purpose        | Authorization   | Authentication + Authorization |
	| Main Token     | Access Token    | ID Token + Access Token        |
	| User Identity  | ❌ Not defined  | ✅ Standardized                |
	| Login Use Case | ❌ Not ideal    | ✅ Designed for login          |
	| Protocol       | Framework       | Protocol built on OAuth2       |
	| Scope          | resource access | identity (`openid`)            |

System Design :
---------------
Frontend (React)
API Gateway
Auth Service
Multiple microservices

Without OIDC:
We manage login manually.
Security risk increases.

With OIDC (Recommended):
Keycloak
Okta
Auth0

Flow:
1.User logs in via OIDC provider.
2.Provider returns ID Token.
3.API Gateway validates token.
4.Pass user identity to services.

OIDC Flows :
------------
Similar to retrieving access tokens in OAuth2.0, there are multiple flows for getting an ID token.
The three that exist are
1.Authorization Code Flow.
2.Implicit Flow
3.Hybrid Flow.
