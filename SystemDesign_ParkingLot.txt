############################################################## 1.Clarify Requirements ############################################################################

1.Functional requirement
2.Non Functional requirement
3.Constraints
4.Out of scope

1.Functional requirement :
==========================
Functional requirements answers what should the system do.
Functional requirements are the Business features of the system.
If this system is built and working then what are the exact operations it should support i.e. visible action that the User or Business expects.

The possible Functional requirements are :
1.The system should allow a vehicle to enter the parking lot.
2.The system should identify the vehicle type such as bike, car, truck, or electric vehicle.
3.The system should assign a suitable parking spot based on vehicle type.
4.The system should generate a parking ticket at entry.
5.The system should store entry time.
6.The system should show available parking spots.
7.The system should allow a vehicle to exit.
8.The system should calculate the parking fee based on entry and exit time.
9.The system should support payment.
10.The system should free the parking spot after successful exit.
11.The system should allow the admin to add or remove parking spots.
12.The system should allow the admin to view occupancy status.

Who are the Users.
What User and Admin can do.
What Business operations must happen.
What Input does the system take.
What Output does the system produce.

The final statement would be "System should assign an available parking slot based on vehicle type".

Note :
------
Functional requirement is the most first important step because jumping directly into architecture often leads to solving the wrong problem.

2.Non Functional requirement :
==============================
Non Functional requirements answers “How well should the system work” where as Functional requirement says what the system does.
Design changes based on Non Functional needs which directly influences the Architecture.

Functional = feature
Non Functional = quality of that feature

Common Non Functional requirements :
------------------------------------
1.Performance
2.Availability
3.Reliability
4.Scalability
5.Security
6.Consistency
7.Maintainability
8.Fault tolerance
9.Observability

1.Performance :
---------------
The system should respond fast because vehicles cannot wait for 2 minutes at the gate.

When the Vehicle enters the Parking Lot :
Lookup for the parking spot based on Vehicle Type should be very fast.
Ticket Generation should be very fast.
Exit Fee calculation should bevery fast.

2.Availability :
----------------
The system should remain up and accessible most of the time.

The parking lot should not stop functioning because of one service crashed.
If the system goes down, vehicles cannot enter or exit.

3.Reliability :
---------------
System should produce correct behavior consistently.

System should not assign the same spot to two cars.
Same car should not get two active tickets accidentally.
Payment should not be marked successful when it actually failed.

4.Scalability :
---------------
System should continue working properly even when the data grows i.e. handle the load.

Small apartment parking may have 50 spots.
Airport parking may have 10,000 spots.
One mall may have 3 gates and another may have 30 gates.

5.Security :
------------
System should protect the data and should not allow the actions which leads to misuse.

Only admin should change parking rates.
Payment details should be protected.
Unauthorized users should not manipulate spot status.

6.Consistency :
---------------
Data should remain correct across the system.

If parking spot becomes occupied :
Database should be reflected as occupied.
Display board should also show the parking spot status as occupied.
Admin dashboard should show the correct count.
One spot should not be shown as free and occupied at the same time in critical operations.

7.Maintainability :
-------------------
System should be easy to update without breaking the existing functionality.

Later we may add multiple parking lots.
Later we may add EV slots.
Later we may add dynamic pricing.
Later we may add reservation feature.

8.Fault tolerance :
-------------------
System should handle the failures gracefully.
Fault tolerance means System should continue working fully or partially even when some of the services fails.

What If network to one gate goes down.
What If payment service is down.
What If the dashboard fails to display the data.

3.Constraints :
===============
A constraint is something that restricts our design choices.
Within the limitations or boundaries we shoud work.
Without constraints we may design an unrealistic solution.
Good system design is not about the most advanced design and It is about the most suitable design under the given constraints.

Constraint tells :
1.what are the existing rules.
2.what environment we must support.
3.what we cannot do.

Functional requirement says need payment where as Constraint says Payment must use existing bank gateway only.
Functional requirement says need Parking Spot assignment where as Constraint says Each gate terminal has limited hardware and poor internet.

A parking lot system may have constraints like :
1.There are already fixed numbers of floors and spots.
2.Existing gate hardware must be reused.
3.Payment must integrate with a specific payment provider.
4.System must support offline ticketing for short network outages.
5.Budget is limited.
6.Only one database is allowed initially.
7.Real-time display boards are already installed and must be supported.
8.Spot assignment must happen in under 2 seconds.
9.One spot can hold only one active vehicle at a time.

Types of Constraints :
1.Business constraints
2.Technical constraints
3.Time constraints
4.Cost constraints

1.Business constraints :
------------------------
Rules from business :
VIP spots cannot be assigned to normal vehicles.
Truck parking allowed only on ground floor.
Parking fee rules are fixed by management.

2.Technical constraints :
-------------------------
Technology-related limitations :
Gates communicate through local network only.
Legacy scanners must be reused.
Existing system uses PostgreSQL.
Do we need to use existing Technology.
No cloud deployment allowed.

3.Time constraints :
--------------------
Project deadline limitations :
Need MVP in 2 months.
The first version should support only one parking lot location.

4.Cost constraints :
--------------------
Cannot use expensive enterprise systems initially.
Prefer monolith instead of microservices for first release.
The initial budget is limited so a simple deployable solution is preferred.

4.Out of scope :
================
Out of scope means what are we intentionally not building right now.
This is extremely important because every system can keep growing forever.
If we do not define what is excluded then the design becomes too large and confusing.

Out of scope does not mean unimportant and It only means
Not needed now.
Not in current version.
Not part of current design exercise.

Architect always reduce ambiguity by saying "We will not cover in this version".

Suppose someone says design a parking lot then we may start thinking about
1.Multi-city centralized management is out of scope.
2.Mobile application for customers is out of scope.
3.Advance booking or Reservation of Parking spots is out of scope.
4.Valet parking support is out of scope.
5.AI camera-based vehicle number plate recognition is out of scope.
6.Dynamic pricing based on demand is out of scope.
7.Loyalty points and offers is out of scope.
8.EV charging workflow is out of scope.
9.Lost ticket workflow is out of scope.

All these may be nice but we are intentionally not handling these now.


############################################################## 2.Define Scope and Assumptions ############################################################################

After Clarifying the requirements we usually have too many possibilities.

1.Multi-city centralized management.
2.Mobile application.
3.Advance booking or Reservation of Parking spots.
4.Valet parking.
5.AI camera-based vehicle number plate recognition.
6.Dynamic pricing based on demand.
7.Loyalty points and offers.
8.EV charging workflow.
9.Lost ticket workflow.

If we don’t limit the problem then our design becomes
❌ Too big
❌ Unclear
❌ Unrealistic

So Defining Scope and Assumptions is about reducing the ambiguity and fixing the problem with clear boundary.
Scope is nothing but What are we building in this version and Assumptions are available defaults which will simplify the design.

We never build everything at once and we always define MVP (Minimum Viable Product).
Defining Scope and Assumptions is defining MVP.

Parking Lot - Scope :
---------------------
1.Single parking lot system.
2.Supports Multi floor.
3.Multiple entry and exit gates.
4.Vehicle types: Bike,Car and Truck.
5.Automatic parking spot allocation.
6.Ticket generation at entry.
7.Fee calculation at exit.
8.Payment handling.
9.Spot release after exit.
10.Display available spots.

Parking Lot - Assumptions :
---------------------------
Without Assumptions we will keep asking questions forever and the Design will never start.
With Assumptions we move forward with clarity.
System design becomes manageable only after assumptions are frozen.
This prevents over-design and keeps the design focused.

1.Parking Lot Structure
Parking lot has multiple floors.
Each floor has fixed number of spots.
Each spot has a predefined type.

2.Spot Rules
One spot = one vehicle.
A spot can be AVAILABLE and OCCUPIED.
No sharing of spots.

3.Vehicle Rules
Each vehicle has a type.
Vehicle type determines allowed spots.

4.Ticket Rules
Each vehicle gets exactly one ticket at entry.
A ticket remains active until exit.
Ticket contains entry time,spot id and vehicle type.

5.Payment Rules
Payment happens only at exit.
Payment must succeed before exit.
Pricing is time-based (simple).

6.Availability Rules
System keeps count of available spots.
Display board shows near real-time data.

7.Concurrency Simplification
Multiple gates can operate simultaneously.
System must prevent double allocation.

8.Deployment Scope
Single location system.
Not distributed across cities.
No multi-region complexity.

9.User Interaction Assumption
Gate system handles entry/exit
No mobile app in current version.

#################################################################### 3.Estimate Scale ####################################################################################

Scale is important in System Design because it affects
1.Architecture.
2.Database choice.
3.Concurrency handling.
4.Caching.
5.Performance expectations.
6.cost.

Before designing architecture we must know the expected size.
So even If the system looks simple we still estimate scale.
Estimate scale means we are trying to understand how load affects design.
How many users or requests will come.
How much data will be stored.
How fast system should respond.
How much growth system should handle.

The design for a small apartment parking lot and a Large airport parking system will not be the same.

Small Parking Lot :
-------------------
100 spots.
1 or 2 gates.
Low traffic.
Simple database enough.

Large parking Lot :
-------------------
Many Floors.
10,000 spots.
20 entry/exit gates.
High traffic.
Real-time occupancy updates needed.
Stronger concurrency control needed.

Advantage :
-----------
1.Is a simple monolith enough.
2.Is relational DB enough.
3.Do we need caching.
4.How much concurrency do we expect.

Parking Lot Scale Estimation :
==============================
Consider the usecase of Large commercial Parking Lot.

1 parking lot location
5 floors
1000 spots per floor i.e. 5000 spots
10 entry gates
10 exit gates

1.Estimate number of vehicles :
-------------------------------
If there are 5000 spots which has the full capacity then maximum 5000 vehicles can be parked at a time.

2.Estimate entry traffic :
--------------------------
How many vehicles may enter in peak time.
Consider around 1200 vehicles enter in 1 hour.

	1200 / 60 = 20 vehicles per minute
	20 / 60 = around 0.33 vehicles per second
	
This still does not look huge.
If the traffic is bursty then at one moment, multiple gates may receive requests together.
Consider across all gates we may recieve

	5 to 10 entry requests per second 
	
3.Estimate exit traffic :
-------------------------
Exit traffic may also spike.
For Example Office closing time, Movie show ends and Mall closes.
Exit flow also includes Fee calculation, Payment and Gate release.
So exit flow may be heavier than entry flow.
Consider around 1800 vehicles exit in 1 hour.

	30 vehicles per minute
	0.5 vehicles per second average
	
With burst

	10 to 15 exit requests per second
	
4.Estimate availability queries :
---------------------------------
Availability data may be requested by
Entry display boards
Admin dashboard

Suppose display boards refresh every 5 seconds
10 entry boards
10 admin/operator screens

	20 / 5 = 4 queries per second

If many systems check availability frequently then it can grow.
So this data may later be cached.

5.Estimate ticket generation :
------------------------------
Each entry creates one ticket.
If 15,000 vehicles use the parking lot in a day then 15,000 tickets/day.
That means 15,000 write operations for entry sessions.
Not very large for a normal database.

6.Estimate payments :
---------------------
Each exit usually creates one payment record.
So If 15,000 vehicles exit per day then 15,000 payments/day.
This is less about scale and more about consistency and reliability.

7.Estimate storage :
--------------------
Suppose each ticket record stores
ticket id
vehicle number
vehicle type
spot id
entry time
exit time
fee
status

Assume one ticket record roughly takes 500 bytes to 1 KB including indexes and metadata.

	15,000 KB/day = 15 MB/day at 1 KB each
	15 MB × 365 ≈ 5.4 GB/year

Even if we add payment info, logs and indexes then maybe 

	10–20 GB/year operational data
	
This is still manageable in relational DB.

8.Estimate concurrent operations :
----------------------------------
Even If the daily data is small, concurrent operations can create problems.
1.Two gates trying to assign same spot.
2.Payment retry happening twice.
3.One vehicle scanned twice.
4.Availability count getting stale.

10–20 concurrent gate operations at peak
many spot updates happening at same time
simultaneous reads for availability

9.Latency expectations :
------------------------
Entry gate :
Driver should not wait long.
Expected response: preferably under 1–2 seconds

Exit gate :
Should also be quick: Fee calculation + payment initiation should be fast.

Availability display :
Should be near real-time: A few seconds delay may be acceptable.

This helps later when deciding :
cache
Precomputed counts
sync vs async updates

Conclusions :
=============
1.We do not need a super-complex internet-scale architecture, simple and clean architecture may be enough.
Not required
Dozens of microservices.
Distributed NoSQL by default.
Huge event streaming platform at day one.

2.Relational database is a good fit.
Data is structured.
Transactions matter.
Payment correctness matters.
Spot assignment correctness matters.

3.Concurrency matters more than huge throughput.
Main risk is
Same spot assigned twice.
Duplicate ticket.
Wrong availability count.
Duplicate payment.

So later we must focus o:
Transactions.
row locking or optimistic locking.
Idempotency.
State transitions.

4.Cache may help availability reads.
Because availability count may be read frequently.
DB remains source of truth.
Cache stores live counts.
Display boards read from cache.

5:Peak load matters more than average load.
Many vehicles enter/exit together. So architecture should be designed for burst handling.

General estimation for any System Design :
==========================================
1.Number of entities or Entity scale.
2.Request scale.
3.Storage scale.
4.Read vs write pattern or Concurrency scale.
5.Peak traffic or Latency expectations.

#################################################################### 4.Identify core use cases and workflows ###############################################################

Use case is a complete business scenario.
A workflow is the step-by-step sequence inside an use case.

Use case = overall scenario.
Workflow = detailed steps in that scenario.

Identifying the Usecases and Core workflows means describing how users and systems interact step by step to achieve the business goals.

Actors in Parking Lot System :
==============================
Before writing use cases, identify the Actors.
An actor is a person, device or an external system that interacts with our system.

Primary actors :
----------------
Vehicle oowner
Parking Attendant
Admin

System actors :
---------------
Entry gate machine
Exit gate machine
Payment gateway
Display board
Sensor / scanner

Main use cases for Parking Lot :
================================
Vehicle enters parking lot
System allocates parking spot
Ticket is generated
Vehicle exits parking lot
Parking fee is calculated
Payment is processed
Spot is released
Availability is updated/displayed
Admin manages parking configuration

1.Vehicle Entry :
=================
Goal :
------
Allow a vehicle to enter the parking lot.

Primary actor :
---------------
Driver, Entry Gate Machine.

Why this Use case exists :
--------------------------
Because every parking session starts here.

Workflow :
----------
1.Vehicle arrives at entry gate.
2.Entry gate machine detects vehicle.
3.System checks whether parking is available for that vehicle type.
4.If space is available then system proceeds.
5.Ticket is generated.
6.Entry time is recorded.
7.Assigned spot details may be shown.
8.Gate opens.
9.Vehicle enters.

Business rules :
----------------
Entry should not be allowed If the parking is full.
Vehicle should get only one active ticket.
Ticket must be tied to one parking session.

Data :
------
Ticket.
Entry timestamp.
Vehicle session.
Maybe provision for spot reservation.

Failure scenario :
------------------
No available spot.
Duplicate scan.
Printer not working.
Database error.
Gate fails to open.

2.Spot Allocation :
===================
Goal :
------
Assign a suitable parking spot to the vehicle.

Primary actor :
---------------
System / Allocation Logic.

Why this Use case exists :
--------------------------
Because vehicle must know where to park and system must track occupancy correctly.

Workflow :
----------
1.System identifies vehicle type.
2.System fetches available spots of matching type.
3.System selects one suitable spot.
4.Spot is marked as reserved or occupied.
5.Spot is linked to the active ticket.
6.Driver is informed of the allocated spot.

Business rules :
----------------
One spot can hold only one active vehicle.
Spot type must match vehicle type.
Same spot should not be assigned twice.

Data :
------
Spot status is updated.
Ticket is created and Spot is assigned.
Availability count is updated.

Failure scenario :
------------------
No matching spot available.
Two gates trying to assign same spot.
Spot state outdated.
Cache says free and DB says occupied.

3.Entry time is saved.
4.Vehicle type and spot details are attached.
5.Ticket status is marked active.
6.Ticket is printed or displayed.

Business rules :

Postconditions :

4.Vehicle Exit :
================
Primary actor :
---------------
Driver, Exit Gate Machine.

Goal :
------
Allow a vehicle to leave parking lot correctly.

Preconditions :
---------------

Why this Use case exists :
--------------------------
Because parking session should end in a controlled and billable way.

Workflow :
----------
1.Vehicle comes to exit gate.
2.Driver provides ticket.
3.System fetches active parking session.
4.System validates ticket.
5.System calculates duration.
6.System calculates parking charge.
7.Payment is requested.
8.On successful payment, exit is approved.
9.Gate opens.
10.Spot is released.
11.Ticket is closed.

Business rules :
----------------
Exit should be based on valid active ticket.
Closed ticket should not be reused.
Spot should be freed only once.
Ticket must move from ACTIVE to CLOSED.

Postconditions :
----------------

Data :
------

Failure scenario :
------------------
Ticket not found.
Ticket already closed.
Wrong ticket used.
Exit attempted without payment.
Payment succeeds but gate does not open.

===================

---------------
Billing/Rate Engine

Goal :
------
Calculate parking charges.

Preconditions :
---------------

Why this Use case exists :

2.System gets current time at exit.
3.Duration is calculated.
4.Pricing rule is applied.
5.Final amount is returned.

Business rules :

Postconditions :
----------------

Data :
------

Timezone mismatch.

---------------
System

------

---------------

--------------------------

----------

2.Associated spot is identified.
3.Spot status changes from OCCUPIED to AVAILABLE.

exit completed twice.
