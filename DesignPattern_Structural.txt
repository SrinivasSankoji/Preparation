####################################### 1.Adapter (Satyendra Jaiswal,Palmurugan C) ###################################################################

Adapter Design Pattern is a structural design pattern that allows two incompatible interfaces to work together.
Adapter acts like a translator between two objects. 
The client keeps using the interface it already understands while the Adapter converts that call into the format expected by the existing or third-party class.
Adapter is used when we cannot change the Existing class.
The Adapter Design Pattern is applicable in scenarios where
1.We need to use an existing class whose interface is different from the interface expected by the client.

	    Client
		   |
		   v
		Target Interface
		   |
		   v
		Adapter
		   |
		   v
		Adaptee (Existing Class)

Usage :
-------
1.We already have a class or third-party library, but its interface does not match what our application expects.
2.We want to connect legacy code with new code.
3.We want to avoid changing the client code again and again for each external system.

Problem :
---------
Sometimes two classes are useful but they cannot directly work together because their method names, parameters or data formats are different.

Example:
One system sends XML.
Other system accepts JSON.
Both systems are fine but they are incompatible.

Solution :
----------
Create an Adapter class.
This adapter receives the request in the format expected by the client.
Converts or translates it.
Calls the actual existing service.
Returns the result in a usable way.
So the client talks to the Adapter and the Adapter talks to the real service.

Participants in Adapter Pattern :
---------------------------------
1.Target 
2.Adapter
3.Adaptee

1.Target :
----------
The interface that the client expects to use.
In simple words Target is the standard format our application already understands.
The client code is written against this interface.
In the weather example LegacyWeatherService is the Target because the client or third-party module expects that interface.

2.Adapter :
-----------
A class that implements the Target interface and internally uses the Adaptee.
In simple words Adapter is the translator.
Adapter accepts the client request in one format and converts it into another format that the adaptee understands.

Main responsibility of Adapter class :
1.Implements Target
2.Holds reference of Adaptee.
3.Converts request.
4.Delegate calls to Adaptee.

WeatherServiceAdapter is the Adapter.
It allows code written for LegacyWeatherService work with ModernWeatherService.

3.Adaptee :
-----------
Adaptee is the existing class whose interface is incompatible with the client.
In simple words Adaptee is the actual class we want to use but we cannot use directly.

Adaptee may be:
1.Legacy code.
2.Third-party SDK.
3.External service client.
4.Newly introduced service with different method signature.

Without Adaptor Design Pattern :
--------------------------------
Client expects  

	void getWeather(String city);
	
But new service provides

	void fetchWeather(String city, double latitude, double longitude);

Code without Adapter :
----------------------

	interface LegacyWeatherService {
		void getWeather(String city);
	}

	class LegacyWeatherServiceImpl implements LegacyWeatherService {
		@Override
		public void getWeather(String city) {
			System.out.println("Legacy Weather Service: Getting weather for " + city);
		}
	}

	class ModernWeatherService {
		public void fetchWeather(String city, double latitude, double longitude) {
			System.out.println("Modern Weather Service: Getting weather for " +
					city + " at latitude " + latitude + " and longitude " + longitude);
		}
	}

	public class WeatherClientWithoutAdapter {
		public static void main(String[] args) {
			LegacyWeatherService legacy = new LegacyWeatherServiceImpl();
			legacy.getWeather("New York");

			// direct usage of modern service
			ModernWeatherService modern = new ModernWeatherService();
			modern.fetchWeather("San Francisco", 0.0, 0.0);

			// third-party module cannot use legacy abstraction cleanly
			ThirdPartyModule module = new ThirdPartyModule();
			module.requestWeather("Los Angeles");
		}
	}
	
	class ThirdPartyModule {
		public void requestWeather(String city) {
			// third-party/client now directly depends on concrete logic
			ModernWeatherService modernWeatherService = new ModernWeatherService();

			// client itself is forced to know extra parameters
			modernWeatherService.fetchWeather(city, 0.0, 0.0);
		}
	}
	
With Adapter Design Pattern :
-----------------------------

Target (expected by client) :
-----------------------------

	interface LegacyWeatherService {
		void getWeather(String city);
	}

	class LegacyWeatherServiceImpl implements LegacyWeatherService {
		@Override
		public void getWeather(String city) {
			System.out.println("Legacy Weather Service: Getting weather for " + city);
		}
	}

Adaptee (Existing incompatible class) :
---------------------------------------

	class ModernWeatherService {
		public void fetchWeather(String city, double latitude, double longitude) {
			System.out.println("Modern Weather Service: Getting weather for " +
					city + " at latitude " + latitude + " and longitude " + longitude);
		}
	}

Adapter :
---------

	class ModernWeatherServiceAdapter implements LegacyWeatherService {

		private final ModernWeatherService modernWeatherService;

		public ModernWeatherServiceAdapter(ModernWeatherService modernWeatherService) {
			this.modernWeatherService = modernWeatherService;
		}

		@Override
		public void getWeather(String city) {
			modernWeatherService.fetchWeather(city, 0.0, 0.0);
		}
	}

	
Client :
--------
	
	class ThirdPartyModule {
		public void requestWeather(LegacyWeatherService service, String city) {
			service.getWeather(city);
		}
	}
	
Execution :
-----------
	
	public class WeatherClientWithAdapter {
		public static void main(String[] args) {
			LegacyWeatherService legacy = new LegacyWeatherServiceImpl();
			legacy.getWeather("New York");

			ModernWeatherService modern = new ModernWeatherService();
			LegacyWeatherService adapter = new ModernWeatherServiceAdapter(modern);
			adapter.getWeather("San Francisco");

			ThirdPartyModule module = new ThirdPartyModule();
			module.requestWeather(adapter, "Los Angeles");
		}
	}

####################################### 3.Composite (Salitha Chathuranga & Abdulaziz Pulatjonov) ########################################

Composition represents a group of Objects defined through a structure or hierarchy.
Composite design pattern allows us to treat individual Objects and composition of Objects in the same way.
Tree structure could be the perfect example for Composite design pattern as it inherits the base type and can be represented as a part of whole hierarchy.
Composite design pattern can be classified into 4 types.

1.Component
2.Leaf
3.Composite
4.Client

1.Component :
-------------
Component is an interface or abstract class with the methods which has to be implemented in its Child composites.
Component is considered as the parent for all the composite Objects.

2.Leaf :
--------
Leaf is an Object which implements the behavior of the Component and does not contain any references to other Objects.             

3.Composite :
-------------
Composite is an Object which implements the behavior of the Component part and contains the references to other composite objects.
Composite is similar to Leaf but the only difference is it references to other Composite or Leaf Objects.

4.Client :
----------
Client is an instance of an Object which has access to composition elements by using Component Object.

Consider the scenario of Employee and Company.

Employee is the Component.
CEO,Manager,QA,Developer are the Leaf objects which does not contain any reference to other Objects.
CompanyDirectory is a composite object which implements the Component part and contains the reference to other Composite or Leaf objects.
Contains the reference of adding or removing the Composite or Leaf objects.

CEO.java :
----------

	public class CEO implements  Employee{
    private final String name;
    private final String dept;
    private final String job;

    public CEO(String name, String dept, String job) {
        this.name = name;
        this.dept = dept;
        this.job = job;
    }

    @Override
    public void showDetails() {
        System.out.println(this);
    }

    @Override
    public String toString() {
        return "CEO{" +
                "name='" + name + '\'' +
                ", dept='" + dept + '\'' +
                ", job='" + job + '\'' +
                '}';
    }
	}

CompanyDirectory.java :
-----------------------

	public class CompanyDirectory implements Employee {

    private final List<Employee> employees;

    public CompanyDirectory() {
        this.employees = new ArrayList<>();
    }

    public void addEmployee(Employee employee) {
        employees.add(employee);
    }

    public void removeEmployee(Employee employee) {
        employees.remove(employee);
    }

    @Override
    public void showDetails() {
        for (Employee employee:employees){
            employee.showDetails();
        }
    }
	}
	
Company.java :
--------------

	public class Company extends CompanyDirectory {

    private final String name;

    public Company(String name) {
        super();
        this.name = name;
    }

    public void showCompanyName() {
        System.out.println("Company: " + name);
    }
	}
	
CompositeDemo.java :
--------------------

	public class CompositeDemo {
    public static void main(String[] args) {
        CEO ceo = new CEO("Nandini", "COM", "Chief Executive Officer");

        Developer developer1 = new Developer("Srinivas", "ENG", "Project manager");
        Developer developer2 = new Developer("XYZ", "ENG", "Associate Software Engineer");

################################################### 4.Decorator (Salitha Chathuranga & java67) ##########################################

https://salithachathuranga94.medium.com/decorator-design-pattern-in-java-1b0931ead0e4

Decorator Design pattern is a structural design pattern that allows us to add a new functionality to an existing Object without changing its structure.
Decorator Design pattern allows us to do runtime modifications of an Object.
Decorator Design Pattern works by wrapping an object inside another object which has the same interface.
The traditional way of adding new functionality to an existing Object without changing its structure is by using inheritance i.e. at compile time.

Usecase :
---------
Consider the scenario of Pizza order with additional features.
1.Cheese
2.Olives
3.Onion
4.Pepperoni
5.Extra Sauce

❌ Without Decorator : 
----------------------
If we want every combination of these features prebuilt then number of classes required is
2 power 5 = 32 different classes because 

For each feature the pizza can either have it or not → 2 choices per feature
2 × 2 × 2 × 2 × 2 = 32 combinations

	| Features                | Class Needed     |
	| ----------------------- | ---------------- |
	| Cheese                  | Yes              |
	| Olives                  | Yes              |
	| Cheese + Olives         | Yes              |
	| Cheese + Onion          | Yes              |
	| Cheese + Olives + Onion | Yes              |
	| … up to 5 features      | Yes → 32 classes |

This quickly becomes unmanageable.

✅ With Decorator :
-------------------
1 Base component.
5 Decorator classes i.e.one for each feature.
Now we can mix and match any combination at runtime.

########################################### 5.Facade (Abdulaziz Pulatjonov & Prabhu Kumar) ##############################################

https://blog.devops.dev/structural-design-patterns-in-java-facade-design-pattern-a6d2797ed8de

Facade Design Pattern is one of the subtype of structural design pattern.
Facade Design Pattern provides a unified interface to a set of interfaces in the Complex sub system.
Facade Design Pattern provides a high level interface that makes the subsystem easier to use.
Facade Design Pattern provides a simplified interface that the Client can be used to access the System.
Here business logic can be hidden behind the interface so that Client does not aware of implementation.

Facade Design Pattern is appropriate when we have a Complex System and want in a simplified way.
The classes and Objects participating in this pattern are 

Subsystem classes
Subsystem classes forms the complex interface of the existing sub system.
Facade
Facade provides simplified interface which delegates calls the to the Classes in the Sub System.
Client
Client invokes Facade which shields away complex interface from the Client.

########################################### 6.Flyweight (Rajeshvelmani & Abdulaziz Pulatjonov) ###########################################

Flyweight Design Pattern is one of the subtype of of the Structural Design Pattern.
Flyweight Design Pattern suggests minimize the creation of new Objects each time by resusing existing ones.
Flyweight Design Pattern is particularly useful when dealing with large number of similar Objects by improoving performance and reducing memory footprint.
Flyweight Design Pattern optimizes the memory usage by sharing common data between multiple Objects.
Instead of each Object holding its own data,seperates the intrinsic state (shared) and extrinsic state (unique) of an Object.

In Flyweight Design Pattern HashMap is used to store the references to the Object that has been created and and associate them with a unique key.
So whenever new Object is needed,Client just passes the unique property and is checked in HashMap.
If an Object with that unique property already exists then reference to that Object will be returned otherwise new Object is created.

Key features of Flyweight Design Pattern are 

Flyweight Interface :
---------------------
This interface defines the methods that the Flyweight Object must implement.
This interface contains the operations that can be shared and executed by multiple Objects.

Flyweight Factory :
------------------
Flyweight Factory creates and manages Flyweight Objects.
Flyweight Factory ensures that Flyweight Objects are shared and reused across multipe Clients.

Concrete Flyweight :
--------------------
Concrete Flyweight Class represents the shared Flyweight Object.
It stores the intrinsic state and provides an implementation for shared operations.
